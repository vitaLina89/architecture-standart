### FURPS+ Таблица требований для MVP "Онлайн-депозиты"

| Категория | Код | Требование | Комментарий (Обоснование, Архитектурная значимость) |
| :--- | :--- | :--- | :--- |
| **Функциональность<br>(Functionality)** | F1.1 | Сайт должен предоставлять возможность новым клиентам просматривать список доступных депозитов с актуальными ставками. | Новый функционал для сайта. **Архитектурно-значимое:** требует источника данных для ставок. |
| | F1.2 | Интернет-банк должен предоставлять возможность существующим клиентам просматривать список доступных депозитов с актуальными ставками. | Новый функционал для интернет-банка. Использует существующую интеграцию с АБС. |
| | F1.3 | Интернет-банк должен предоставлять возможность существующим клиентам просматривать персонализированные ставки по депозитам, рассчитанные специально для них. | Новый функционал. Требует расчетов на основе данных клиента из АБС. **Архитектурно-значимое:** логика расчета персонализации. |
| | F2.1 | Сайт должен предоставлять возможность новому клиенту подать заявку на депозит, указав номер телефона и ФИО. | Новый функционал для сайта. Требует интеграции с системой кол-центра. |
| | F2.2 | Интернет-банк должен предоставлять возможность существующему клиенту подать заявку на открытие депозита, указав счет и сумму депозита. | Новый функционал для интернет-банка. Требует расширения интеграции с АБС. |
| | F2.3 | Интернет-банк должен требовать подтверждения операции подачи заявки с помощью СМС-кода, отправляемого через существующий СМС-шлюз. | Использование существующего СМС-шлюза. **Архитектурно-значимое:** необходимость доработки/интеграции без участия подрядчика. |
| | F3.1 | Сайт должен передавать заявку на депозит в существующую систему кол-центра для обработки менеджером. | **Архитектурно-значимое:** интеграция сайта (PHP) с системой кол-центра (Java Spring Boot, микросервисы). |
| | F4.1 | АБС должна предоставлять возможность менеджеру бэк-офиса обработать заявку на открытие депозита (ручная обработка в MVP). | АБС уже существует. Нужно добавить функционал обработки заявок. |
| | F4.2 | АБС должна предоставлять возможность менеджеру бэк-офиса подтвердить условия депозита в АБС. | **Архитектурно-значимое:** точка интеграции, где решение из заявки превращается в банковский продукт. |
| | F4.3 | АБС должна предоставлять возможность согласования ставки по заявке на депозит. | Весь процесс согласования ставки должен быть реализован в АБС. **Архитектурно-значимое:** централизация бизнес-логики. |
| | F5.1 | АБС должна предоставлять возможность хранения ставок по депозитам в своей базе данных (вместо XLS-файлов). | **Архитектурно-значимое:** создание единого, надежного источника истины для ставок. Перенос данных из XLS в Oracle. |
| | F6.1 | Система должна отправлять СМС-уведомление клиенту после подтверждения ставки и после открытия депозита. | **Архитектурно-значимое:** требует интеграции АБС с СМС-шлюзом. |
| | F7.1 | Сайт и интернет-банк должны обеспечивать шифрование трафика (HTTPS/TLS) при передаче данных заявки. | Стандартное требование безопасности, но критичное для защиты чувствительных данных. |
| **Удобство использования<br>(Usability)** | U1.1 | Интерфейс должен соответствовать системе дизайна компании, используя привычные цвета и брендированные элементы. | Обеспечивает единообразие и узнаваемость бренда. |
| | U1.2 | Интерфейс должен быть максимально удобным для клиента, обеспечивая интуитивно понятное взаимодействие. | Критично для привлечения клиентов новых категорий. |
| | U2.1 | Интерфейс должен быть адаптирован для пользователей старшего возраста: крупный шрифт, высокая контрастность, простые формы. | **Архитектурно-значимое:** основная текущая категория клиентов; влияет на UI/UX дизайн. |
| | U8.1 | Интерфейс должен корректно отображаться на различных устройствах (десктоп, планшет, мобильный телефон). | **Архитектурно-значимое:** требует адаптивной верстки. |
| **Надежность<br>(Reliability)** | R1.1 | Все сервисы системы должны работать 24/7 и быть доступны в 99,9% случаев. | Требование бизнеса. Критично для банковских систем. **Архитектурно-значимое:** определяет стратегию развертывания и резервирования. |
| | R1.3 | Система должна обеспечивать доступность сервисов даже в случае сбоев в ЦОД, используя резервный ЦОД для переключения. | **Архитектурно-значимое:** требует отказоустойчивой архитектуры развертывания. |
| | R2.3 | Система должна обеспечивать отказоустойчивость интеграций между компонентами с механизмами повторных попыток. | **Архитектурно-значимое:** особенно для связки Интернет-банк -> АБС и АБС -> СМС-шлюз. |
| | R3.2 | Система должна обеспечивать распределение нагрузки между серверами и приложениями с равномерным горизонтальным масштабированием. | Требование для интернет-банка. **Архитектурно-значимое:** противоречит ограничению АБС на вертикальное масштабирование. |
| | R11.1 | Система должна обрабатывать недоступность АБС с информированием пользователя и отложенной обработкой заявки. | **Архитектурно-значимое:** требует введения асинхронного взаимодействия (очереди) для изоляции от сбоев АБС. |
| | R13.1 | Система должна обеспечивать защиту от перегрузки базы данных АБС при большом количестве запросов. | **Архитектурно-значимое:** прямое указание на проблему. Требует буферизации и регулирования запросов к АБС. |
| **Производительность<br>(Performance)** | P1.1 | Время отклика для пользовательских операций не должно превышать 500 мс для 95% запросов. | Требование бизнеса: "отклик по всем операциям должен быть максимально быстрым". |
| | P1.2 | Время загрузки справочных данных (список депозитов) не должно превышать 1 секунду. | Прямое указание на существующую проблему. **Архитектурно-значимое:** требует введения кэширования. |
| | P2.1 | Система должна выдерживать пиковую нагрузку в 100 одновременных сеансов работы с модулем депозитов. | Оценка на основе текущего низкого использования и планов по развитию. |
| | P2.2 | Система должна обрабатывать до 50 заявок на депозит в минуту в пиковом режиме. | Обеспечивает пропускную способность для нового канала притока клиентов. |
| | P3.1 | Запросы к АБС для получения справочной информации (ставки) должны быть кэшированы со временем жизни не более 5 минут. | **Архитектурно-значимое:** ключевая мера для разгрузки АБС и обеспечения производительности. |
| | P3.2 | Запросы от интернет-банка к АБС на создание заявки должны выполняться асинхронно, чтобы не блокировать UI. | **Архитектурно-значимое:** решение для длительных операций и защиты АБС от пиковых нагрузок. |
| **Поддерживаемость<br>(Supportability)** | S1.1 | Разработка должна максимально использовать существующие в банке технологии (MS SQL, Oracle, .NET, Java). | **Архитектурно-значимое:** ограничивает выбор технологического стека для новых компонентов. |
| | S1.2 | Для новых компонентов следует использовать Kafka в качестве брокера сообщений. | **Архитектурно-значимое:** стратегическое решение, несмотря на текущую несовместимость с интернет-банком. |
| | S2.1 | Все изменения должны сопровождаться документацией для дальнейшего расширения системы. | Прямое требование от команды трансформации. |
| | S2.2 | Для нового функционала в интернет-банке следует рассмотреть переход на микросервисную архитектуру (в рамках задачи). | **Архитектурно-значимое:** указание на стратегическое направление развития, влияет на изоляцию сервисов. |
| | S3.1 | Логирование должно быть централизованным и обеспечивать сквозную трассировку заявки по всем системам. | Критично для отладки распределенной системы и анализа инцидентов. |
| | S3.2 | Должны быть реализованы health-checks для всех ключевых сервисов (Интернет-банк, сервис депозитов, интеграционный слой). | Обеспечивает возможность мониторинга и быстрого обнаружения сбоев. |
| **+ Ограничения<br>(Restrictions)** | +R1 | Интернет-банк является монолитом на ASP.NET MVC 4.5 (.NET Framework 4.5) и MS SQL. | **Жесткое ограничение.** Определяет технологию разработки изменений в IB. |
| | +R2 | АБС является десктоп-клиентом на Delphi с бэкендом на PL/SQL в Oracle. Масштабируется только вертикально. | **Критичное архитектурное ограничение.** Определяет необходимость в буферизации и асинхронном взаимодействии. |
| | +R3 | Прямая работа интернет-банка с API АБС в новом процессе нежелательна. | **Архитектурное ограничение.** Требует введения промежуточного слоя (например, очереди или отдельного сервиса). |
| | +R4 | Система кол-центра — внешняя микросервисная система на Java Spring Boot и PostgreSQL. | **Ограничение интеграции.** Определяет протоколы и форматы данных для взаимодействия. |
| | +R5 | СМС-функционал требует доработки со стороны подрядчика, но этого лучше избежать. | **Ограничение.** Подталкивает к реализации силами команды банка, используя существующие API шлюза. |
| | +R6 | Новая функциональность не должна нарушать текущие бизнес-процессы отделений. | **Бизнес-ограничение.** Изменения должны быть обратно совместимыми. |
